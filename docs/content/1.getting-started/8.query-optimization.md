## Overview

Choosing the right data-fetching strategy is critical for Nuxt applications that need to balance **performance** (fast server-side rendering, CDN caching) with **editorial experience** (live preview, Visual Editing overlays). This guide explains the trade-offs of each approach and when to use each one.

---

## The Three Strategies at a Glance

| Composable | Runs on Server? | CDN-cached? | Visual Editing? | Live Updates? |
|---|---|---|---|---|
| `useAsyncData` + `sanity.fetch` | ✅ Yes | ✅ Yes | ❌ No | ❌ No |
| `useSanityQuery` | ✅ Yes (initial) | ⚠️ No (uses auth token in preview) | ✅ Yes | ✅ Yes |
| `useSanitySmartQuery` | ✅ Yes | ✅ Yes | ✅ Yes | ✅ Yes |

---

## Why `useSanityQuery` Is Always Client-Side for Previews

`useSanityQuery` is the right choice for any page that needs Sanity Visual Editing or Presentation Tool support. However, its current implementation captures the client selection at composable setup time. On the **first page load before preview is activated** (e.g. before Presentation Tool has sent a perspective cookie), the composable uses the public CDN client. Once the Presentation Tool activates preview, the perspective cookie is updated, the `useAsyncData` watcher fires, and the composable refetches — but the client switch only happens when the composable is re-initialised (e.g. after a page navigation or a manual `refresh()` call).

**In short:** if your Nuxt page is opened directly inside Sanity Presentation Tool without a prior page navigation, `useSanityQuery` requires one extra navigation before stega overlays appear.

This is an acceptable trade-off for many editorial workflows, but `useSanitySmartQuery` solves it elegantly.

## `useAsyncData` + `sanity.fetch` — Pure SSR

Best for: public pages that are never viewed in Presentation Tool.

```ts
<script setup lang="ts">
const sanity = useSanity()
const query = groq`*[_type == "post" && slug.current == $slug][0]`

const { data: post } = await useAsyncData(
  `post-${route.params.slug}`,
  () => sanity.fetch<Post>(query, { slug: route.params.slug }),
)
</script>
```

**Pros:**
- Fully server-rendered and CDN-cacheable — no authentication token is sent in the request.
- Zero client-side fetching cost on non-editorial visits.

**Cons:**
- No stega encoding, so Visual Editing overlays will not appear.
- No live preview: draft content is never shown.

---

## `useSanityQuery` — Full Visual Editing Support

Best for: any page that may be viewed inside Sanity Presentation Tool or requires live preview.

```ts
<script setup lang="ts">
const query = groq`*[_type == "post" && slug.current == $slug][0]`
const { data: post, encodeDataAttribute } = await useSanityQuery<Post>(
  query,
  { slug: route.params.slug },
)
</script>
```

See [Visual Editing](/getting-started/visual-editing) for the full setup guide.

**Pros:**
- Automatic stega encoding and live content updates when in Presentation Tool.
- Returns `encodeDataAttribute` for rendering Visual Editing attribute helpers.

**Cons:**
- When `previewMode` is enabled, the initial client-side fetch may use an authenticated endpoint, skipping the CDN — even for public page views.
- The initial page visit inside Presentation Tool may not show overlays until after one page navigation (see above).

---

## `useSanitySmartQuery` — Hybrid SSR + Visual Editing

Best for: pages that serve public traffic at scale **and** need to be editable inside Sanity Presentation Tool without a stale initial render.

`useSanitySmartQuery` is a drop-in opt-in alternative to `useSanityQuery` with a single key difference: **the client is selected reactively inside the `useAsyncData` callback**. This means:

1. On SSR the fast public CDN endpoint is always used — no token, no extra latency.
2. On the client when preview is inactive, the hydrated SSR data is used as-is — no extra fetch.
3. When Presentation Tool activates preview (perspective cookie updates), the composable immediately refetches through the authenticated proxy endpoint with stega encoding and live updates.

The API is identical to `useSanityQuery`, so migration is a one-word change:

```diff
-const { data, encodeDataAttribute } = await useSanityQuery<Product>(query, params)
+const { data, encodeDataAttribute } = await useSanitySmartQuery<Product>(query, params)
```

---

### Example: E-commerce Product Page

A product page needs fast SSR for SEO and CDN performance, but content editors must be able to preview and click-to-edit fields in Sanity Presentation Tool.

```ts
<script setup lang="ts">
const route = useRoute()
const query = groq`*[_type == "product" && slug.current == $slug][0]{
  _id,
  title,
  description,
  price,
  images[]{ asset->{ url } },
}`

const { data: product, encodeDataAttribute } = await useSanitySmartQuery<Product>(
  query,
  { slug: route.params.slug },
)
</script>

<template>
  <div v-if="product">
    <!-- Text fields: stega encoding in the string value is enough for overlays -->
    <h1>{{ product.title }}</h1>
    <p>{{ product.description }}</p>

    <!-- Number: stega cannot encode numbers, explicit attribute required -->
    <span :data-sanity="encodeDataAttribute(['price'])">${{ product.price }}</span>

    <!-- Image: stega cannot encode a URL attribute, explicit attribute required -->
    <img
      v-for="image in product.images"
      :key="image.asset.url"
      :src="image.asset.url"
      :data-sanity="encodeDataAttribute(['images', product.images.indexOf(image), 'asset'])"
    />
  </div>
</template>
```

**What happens:**
- Public visitor: page is server-rendered using the CDN endpoint, no token, no extra client fetch.
- Content editor in Presentation Tool: perspective cookie is set → `useSanitySmartQuery` refetches through the proxy endpoint → stega encoding and edit overlays activate automatically.

### Example: Multilingual Site with `@nuxtjs/i18n`

When using [`@nuxtjs/i18n`](https://i18n.nuxtjs.org), the current locale is a reactive ref. Pass it directly as a param — `useSanitySmartQuery` watches it automatically and refetches whenever the user switches language, including live inside Presentation Tool.

```ts
<script setup lang="ts">
// Requires @nuxtjs/i18n installed and configured
const { locale } = useI18n()
const route = useRoute()

const query = groq`
  *[_type == "page" && language == $lang && slug.current == $slug][0]{
    _id,
    title,
    body,
    seo
  }
`

// locale is a Ref<string> — useSanitySmartQuery watches it reactively.
// Switching language re-fetches from CDN in production, from preview proxy
// when inside Presentation Tool.
const { data: page } = await useSanitySmartQuery<Page>(query, {
  lang: locale,
  slug: computed(() => route.params.slug as string),
})
</script>
```

**What happens:**
- `locale` changes (user clicks a language switcher) → `useSanitySmartQuery` refetches with the new `$lang` param.
- In Presentation Tool, the same reactive refetch applies: language switches are reflected live without a full page reload.
- On SSR, the locale from the Nuxt i18n middleware is used immediately — the CDN can vary-cache per `Accept-Language` or per route prefix.

## `useSanityPreviewState` — Building Custom Preview UI

For advanced use cases where you need to react to preview state in your own components (e.g. rendering an "Edit in Studio" badge, or conditionally fetching draft vs published content in a custom way), use `useSanityPreviewState`:

```ts
const { isPreview, perspective, stegaEnabled } = useSanityPreviewState()
```

| Property | Type | Description |
|---|---|---|
| `isPreview` | `ComputedRef<boolean>` | `true` when Visual Editing / Presentation Tool preview is active |
| `perspective` | `ComputedRef<ClientPerspective>` | Current query perspective (`'published'` or `'previewDrafts'` / `'drafts'`) |
| `stegaEnabled` | `ComputedRef<boolean>` | `true` when stega is both configured and currently active |

All values degrade gracefully to safe defaults (`false` / `'published'`) when visual editing is not configured, so `useSanityPreviewState` is safe to call on any page.

```vue
<template>
  <!-- Only visible to content editors inside Presentation Tool -->
  <div v-if="isPreview" class="preview-banner">
    Previewing {{ perspective }} content
  </div>
</template>

<script setup lang="ts">
const { isPreview, perspective } = useSanityPreviewState()
</script>
```

## Summary: When to Use What

| Situation | Recommended composable |
|---|---|
| Public page, no editorial use, maximum CDN performance | `useAsyncData` + `sanity.fetch` |
| Page that will be edited in Presentation Tool | `useSanityQuery` |
| High-traffic public page that **also** needs Presentation Tool editing | `useSanitySmartQuery` |
| Building custom preview UI / conditional rendering | `useSanityPreviewState` |

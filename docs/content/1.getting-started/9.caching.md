## Overview

Deploying a Nuxt + Sanity application at scale requires balancing two competing goals:

- **Maximum performance** for public users: render HTML once, serve it from an edge CDN at near-zero latency for every subsequent request.
- **Maximum editorial experience** for content editors: always fetch fresh, draft-aware content with visual editing overlays active.

These two goals point in opposite directions at the same URL. This guide explains the full architecture needed to achieve both simultaneously.

---

## The Three Caching Layers

Requests to your Nuxt app pass through up to three caches:

```
Incoming request
       │
       ▼
┌─────────────────────┐
│   Edge CDN          │  Vercel Edge Network, Netlify CDN, Cloudflare…
│   (closest to user) │  Global, near-zero latency on cache hit
└─────────┬───────────┘
          │ cache miss or bypass
          ▼
┌─────────────────────┐
│   Nitro             │  routeRules: { swr } / { isr }
│   (Nuxt server)     │  Process-local cache managed by Nitro
└─────────┬───────────┘
          │ cache miss or bypass
          ▼
┌─────────────────────┐
│   Sanity CDN        │  Already handled by useSanitySmartQuery
│   (data layer)      │  published content, globally distributed
└─────────────────────┘
```

Each layer caches aggressively for public users and must be bypassed entirely for editors. This guide shows how to configure all three.

---

## Design Principle: On-Demand ISR over Short TTLs

A common first approach is to set a short `swr` TTL (e.g. 60 seconds) so that stale content expires quickly. This trades performance for freshness, and it does not actually solve the problem: content updated one second after a cache fill will still be stale for 59 seconds.

A better approach is to:

1. Set a **long TTL** (`swr: 86400` — 24 hours) as a safety net.
2. Rely on a **Sanity webhook** to actively invalidate the cache the moment content is published.

This gives CDN-level performance on every request and guaranteed freshness within seconds of publication.

---

## Dual-Path Architecture

```
Public user ──────► Edge CDN hit ──────────────────────────────► cached HTML (fast)
                        │
                    (cache miss or post-webhook invalidation)
                        │
                        ▼
                    Nitro → SSR → Sanity CDN → fresh HTML → stored in both caches

Content editor ───► preview cookie detected
                        │
                        ▼
                    Edge + Nitro bypass ──► SSR → Sanity API (drafts + stega) → fresh HTML (not cached)
```

The key is detecting the preview cookie early enough to prevent the response from ever entering any cache.

## Step 1 — Nitro Route Rules

Configure long-lived SWR for all editable routes. Webhook-based invalidation handles freshness; the TTL is just the fallback.

```ts{}[nuxt.config.ts]
export default defineNuxtConfig({
  routeRules: {
    '/products/**': { swr: 86400 },
    '/blog/**':     { swr: 86400 },
    '/pages/**':    { swr: 86400 },
  },
})
```

---

## Step 2 — Preview Cache Bypass Middleware

Add a server middleware that sets `Cache-Control: no-store` whenever the Sanity preview cookie is present. This prevents both Nitro and the upstream edge CDN from storing the response.

```ts{}[server/middleware/preview-cache-bypass.ts]
import { getCookie, setResponseHeader } from 'h3'

export default defineEventHandler((event) => {
  const previewCookie = getCookie(event, 'sanity-preview-id')
  if (!previewCookie) return
  setResponseHeader(event, 'Cache-Control', 'no-store')
})
```

::tip
`sanity-preview-id` is the cookie set by the module's built-in `/preview/enable` endpoint. The module sets this cookie automatically when an editor opens the Presentation Tool.
::

---

## Step 3 — Tag Responses with Sanity Document IDs

To enable targeted cache invalidation (purging only the pages affected by a specific document change), tag each SSR response with the Sanity `_id` of the document it was built from.

The tag is sent as a response header. The name of the header depends on your hosting platform (see [Platform Notes](#platform-notes) below).

Create a server-only composable in your app:

```ts{}[composables/useCacheTag.ts]
import { setResponseHeader } from 'h3'

export const useCacheTag = (tag: string | string[]) => {
  if (!import.meta.server) return
  const event = useRequestEvent()
  if (!event) return
  const value = Array.isArray(tag) ? tag.join(',') : tag
  // Replace the header name with the one for your platform — see Platform Notes
  setResponseHeader(event, 'Cache-Tag', value)
}
```

Call it in your page after fetching the document:

```vue{}[pages/products/[slug].vue]
<script setup lang="ts">
const route = useRoute()
const query = groq`*[_type == "product" && slug.current == $slug][0]`

const { data: product } = await useSanitySmartQuery<Product>(query, {
  slug: route.params.slug,
})

// Tag this response with the document _id so the CDN can purge it on publish
if (product.value?._id) {
  useCacheTag(product.value._id)
}
</script>
```

When the document changes in Sanity, the webhook (Step 4) will purge every cached response that carries that `_id` as a tag — and only those responses.

## Step 4 — Webhook Revalidation Route

Create a server route that Sanity calls whenever a document is published. It must:

1. Verify the webhook secret (prevents unauthorized cache purges).
2. Purge the edge CDN cache for the affected document's tag.
3. Clear Nitro's internal server cache.

```ts{}[server/api/_sanity/revalidate.post.ts]
export default defineEventHandler(async (event) => {
  const body = await readBody(event)
  const config = useRuntimeConfig()

  if (event.headers.get('X-Sanity-Webhook-Secret') !== config.sanityWebhookSecret) {
    throw createError({ statusCode: 401 })
  }

  // Platform-specific edge CDN purge — see Platform Notes below
  await purgeCdnCache(body._id)

  // Nitro internal cache
  const storage = useStorage('cache')
  await storage.clear()

  return new Response('Purged successfully!', { status: 202 })
})
```

::warning
`storage.clear()` removes all Nitro cache entries at once. This is the simplest approach and works well for most sites. For large sites with many independently-cached routes, see [Targeted Nitro Invalidation](#targeted-nitro-invalidation) below.
::

Expose the webhook secret in your runtime config:

```ts{}[nuxt.config.ts]
export default defineNuxtConfig({
  runtimeConfig: {
    sanityWebhookSecret: '', // set via NUXT_SANITY_WEBHOOK_SECRET
  },
})
```

## Targeted Nitro Invalidation

If `storage.clear()` is too broad, you can compute and remove only the cache keys for the routes that depend on the changed document.

Nitro stores route cache entries with keys in the form `nitro:handlers:<encoded-path>`. You can remove specific entries using `storage.removeItem`:

```ts{}[server/api/_sanity/revalidate.post.ts]
const storage = useStorage('cache')

// Example: build the list of affected paths from the document type and slug
const affectedPaths = getAffectedPaths(body._type, body.slug?.current)

await Promise.all(
  affectedPaths.map(path =>
    storage.removeItem(`nitro:handlers:${encodeURIComponent(path)}`)
  )
)

function getAffectedPaths(type: string, slug?: string): string[] {
  if (type === 'product' && slug) return [`/products/${slug}`]
  if (type === 'post' && slug) return [`/blog/${slug}`]
  if (type === 'siteSettings') return ['/']
  return []
}
```

::tip
Refer to the [Nitro cache storage documentation](https://nitro.build/guide/cache) for the full list of available storage keys and methods.
::

## Platform Notes

### Vercel

Vercel uses the `Cache-Tag` response header to associate cached responses with tags, and the [Vercel Data Cache API](https://vercel.com/docs/infrastructure/data-cache) (or `@vercel/functions`) to purge them by tag.

```ts{}[composables/useCacheTag.ts]
setResponseHeader(event, 'Cache-Tag', value)
```

```ts{}[server/api/_sanity/revalidate.post.ts]
import { revalidateTag } from '@vercel/functions'

await revalidateTag(body._id)
```

### Netlify

Netlify uses the `Netlify-Cache-Tag` response header and `purgeCache` from `@netlify/functions`.

```ts{}[composables/useCacheTag.ts]
setResponseHeader(event, 'Netlify-Cache-Tag', value)
```

```ts{}[server/api/_sanity/revalidate.post.ts]
import { purgeCache } from '@netlify/functions'

await purgeCache({ tags: [body._id] })
```

### Other platforms

For Cloudflare Pages/Workers, use the [Cache API](https://developers.cloudflare.com/workers/runtime-apis/cache/) with tagged purge via the Cloudflare API. The `useCacheTag` composable pattern remains the same — only the purge implementation changes.

## Step 5 — Configure the Sanity Webhook

In [sanity.io/manage](https://sanity.io/manage), open your project → **API** → **Webhooks** → **Add webhook**:

| Field | Value |
|---|---|
| **Name** | Nuxt ISR Revalidation |
| **URL** | `https://your-site.com/api/_sanity/revalidate` |
| **Dataset** | your dataset (e.g. `production`) |
| **Trigger on** | Create, Update, Delete |
| **Filter** | _(optional)_ `_type in ["product", "post", "page"]` |
| **HTTP method** | POST |
| **HTTP headers** | `X-Sanity-Webhook-Secret: <your-secret>` |
| **Projection** | `{ _id, _type, slug }` |

Set `NUXT_SANITY_WEBHOOK_SECRET` in your hosting environment to the same secret value.

## Completing the Picture: Live Content

The webhook + cache invalidation strategy ensures **new requests** always get fresh HTML. But what about users who already have the page open in their browser?

This is where `liveContent` comes in. When enabled, the module subscribes to Sanity's Live Content API on the client. When a document changes, the subscription fires, `refreshNuxtData` is called, and the page updates — without a full reload.

```ts{}[nuxt.config.ts]
export default defineNuxtConfig({
  sanity: {
    liveContent: {
      browserToken: process.env.NUXT_SANITY_LIVE_BROWSER_TOKEN,
    },
  },
})
```

Together:

| Mechanism | What it keeps fresh |
|---|---|
| Webhook → cache purge | New SSR requests after a publish event |
| `liveContent` | Browser tabs already open when content changes |
| Preview bypass middleware | Editor sessions — always bypass, always fresh |

See the [Live Content guide](/getting-started/live-content) for full setup instructions.

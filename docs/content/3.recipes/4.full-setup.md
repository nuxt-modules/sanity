## Cached Page + Visual Editing

The complete production setup: ISR caching for public users with webhook invalidation, and full visual editing for content editors. Combines the [Cached page](/recipes/cached-page) and [Page with visual editing](/recipes/visual-editing-page) recipes.

---

## When to use this

You are building a production site where public pages must be fast (CDN-cached HTML) and content editors need live preview with click-to-edit overlays inside the Presentation Tool.

---

## How the layers interact

```
Public user ──────► Edge CDN hit ────────────────────────────► cached HTML (fast)
                        │
                    (cache miss or post-webhook invalidation)
                        │
                        ▼
                    Nitro → SSR → Sanity CDN → fresh HTML → stored in both caches

Content editor ───► preview cookie detected
                        │
                        ▼
                    Edge + Nitro bypass ──► SSR → Sanity API (drafts + stega) → fresh HTML (not cached)
```

The preview bypass middleware is the key: it emits `Cache-Control: no-store` for any request carrying the Sanity preview cookie, preventing the editor's draft-aware response from ever polluting the public cache.

---

## Files

- `nuxt.config.ts` — route rules, visual editing config, webhook secret
- `server/middleware/preview-cache-bypass.ts` — cache bypass for editors
- `server/api/_sanity/revalidate.post.ts` — webhook handler
- `pages/[slug].vue` — the page component

## Implementation

```ts{}[nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@nuxtjs/sanity'],
  routeRules: {
    '/pages/**': { swr: 86400 },
    '/blog/**':  { swr: 86400 },
  },
  sanity: {
    projectId: process.env.NUXT_SANITY_PROJECT_ID,
    dataset: process.env.NUXT_SANITY_DATASET,
    visualEditing: {
      token: process.env.NUXT_SANITY_VISUAL_EDITING_TOKEN,
      studioUrl: process.env.NUXT_SANITY_VISUAL_EDITING_STUDIO_URL,
    },
  },
  runtimeConfig: {
    sanityWebhookSecret: '', // set via NUXT_SANITY_WEBHOOK_SECRET
  },
})
```

```ts{}[server/middleware/preview-cache-bypass.ts]
import { getCookie, setResponseHeader } from 'h3'

export default defineEventHandler((event) => {
  const previewCookie = getCookie(event, 'sanity-preview-id')
  if (!previewCookie) return
  setResponseHeader(event, 'Cache-Control', 'no-store')
})
```

```ts{}[server/api/_sanity/revalidate.post.ts]
export default defineEventHandler(async (event) => {
  const body = await readBody(event)
  const config = useRuntimeConfig()

  if (event.headers.get('X-Sanity-Webhook-Secret') !== config.sanityWebhookSecret) {
    throw createError({ statusCode: 401 })
  }

  // Platform-specific edge CDN purge — see caching guide for Vercel / Netlify examples
  await purgeCdnCache(body._id)

  // Clear Nitro's internal server cache
  const storage = useStorage('cache')
  await storage.clear()

  return new Response('Purged successfully!', { status: 202 })
})
```

```vue{}[pages/[slug].vue]
<script setup lang="ts">
const route = useRoute()

const query = groq`*[_type == "page" && slug.current == $slug][0]{
  _id,
  title,
  body,
}`

const { data: page } = await useSanityQuery<Page>(query, { slug: route.params.slug })
</script>

<template>
  <div v-if="page">
    <h1>{{ page.title }}</h1>
    <SanityContent :value="page.body" />
  </div>
</template>
```

## Notes

::tip
The route rules (`swr: 86400`) and the bypass middleware work independently. Route rules apply to all requests by default; the middleware overrides `Cache-Control` only when the preview cookie is present — the two never conflict.
::

::tip
`useSanityQuery` handles the client selection automatically: public CDN on SSR, draft-aware proxy when inside Presentation Tool. No conditional logic is needed in your component.
::

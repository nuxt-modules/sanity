## Overview

Choosing the right data-fetching strategy is critical for Nuxt applications that need to balance **performance** (fast server-side rendering, CDN caching) with **editorial experience** (live preview, Visual Editing overlays). This guide explains the trade-offs of each approach and when to use each one.

---

## `useAsyncData` + `sanity.fetch` ‚Äî Pure SSR

Best for: public pages that are never viewed in Presentation Tool.

```ts
<script setup lang="ts">
const sanity = useSanity()
const query = groq`*[_type == "post" && slug.current == $slug][0]`

const { data: post } = await useAsyncData(
  `post-${route.params.slug}`,
  () => sanity.fetch<Post>(query, { slug: route.params.slug }),
)
</script>
```

**Pros:**
- Fully server-rendered and CDN-cacheable ‚Äî no authentication token is sent in the request.
- Zero client-side fetching cost on non-editorial visits.

**Cons:**
- No stega encoding, so Visual Editing overlays will not appear.
- No live preview: draft content is never shown.

---

## `useSanityQuery` ‚Äî Full Visual Editing Support

Best for: any page that may be viewed inside Sanity Presentation Tool or requires live preview.

```ts
<script setup lang="ts">
const query = groq`*[_type == "post" && slug.current == $slug][0]`
const { data: post, encodeDataAttribute } = await useSanityQuery<Post>(
  query,
  { slug: route.params.slug },
)
</script>
```

See [Visual Editing](/getting-started/visual-editing) for the full setup guide.

**Pros:**
- Automatic stega encoding and live content updates when in Presentation Tool.
- Returns `encodeDataAttribute` for rendering Visual Editing attribute helpers.
- Client and stega selection are evaluated on every refetch, so a perspective change mid-session (e.g. Presentation Tool activating preview) immediately switches to the authenticated proxy endpoint without requiring a page navigation.

**Cons:**
- When `previewMode` is enabled, the initial client-side fetch may use an authenticated endpoint, skipping the CDN ‚Äî even for public page views.

### Example: E-commerce Product Page

A product page needs fast SSR for SEO and CDN performance, but content editors must be able to preview and click-to-edit fields in Sanity Presentation Tool.

```ts
<script setup lang="ts">
const route = useRoute()
const query = groq`*[_type == "product" && slug.current == $slug][0]{
  _id,
  title,
  description,
  price,
  images[]{ asset->{ url } },
}`

const { data: product, encodeDataAttribute } = await useSanityQuery<Product>(
  query,
  { slug: route.params.slug },
)
</script>

<template>
  <div v-if="product">
    <!-- Text fields: stega encoding in the string value is enough for overlays -->
    <h1>{{ product.title }}</h1>
    <p>{{ product.description }}</p>

    <!-- Number: stega cannot encode numbers, explicit attribute required -->
    <span :data-sanity="encodeDataAttribute(['price'])">${{ product.price }}</span>

    <!-- Image: stega cannot encode a URL attribute, explicit attribute required -->
    <img
      v-for="image in product.images"
      :key="image.asset.url"
      :src="image.asset.url"
      :data-sanity="encodeDataAttribute(['images', product.images.indexOf(image), 'asset'])"
    />
  </div>
</template>
```

### Example: Multilingual Site with `@nuxtjs/i18n`

When using [`@nuxtjs/i18n`](https://i18n.nuxtjs.org), the current locale is a reactive ref. Pass it directly as a param ‚Äî `useSanityQuery` watches it automatically and refetches whenever the user switches language, including live inside Presentation Tool.

```ts
<script setup lang="ts">
// Requires @nuxtjs/i18n installed and configured
const { locale } = useI18n()
const route = useRoute()

const query = groq`
  *[_type == "page" && language == $lang && slug.current == $slug][0]{
    _id,
    title,
    body,
    seo
  }
`

// locale is a Ref<string> ‚Äî useSanityQuery watches it reactively.
// Switching language re-fetches from CDN in production, from preview proxy
// when inside Presentation Tool.
const { data: page } = await useSanityQuery<Page>(query, {
  lang: locale,
  slug: computed(() => route.params.slug as string),
})
</script>
```

## `useSanityPreviewState` ‚Äî Building Custom Preview UI

For advanced use cases where you need to react to preview state in your own components (e.g. rendering an "Edit in Studio" badge, or conditionally fetching draft vs published content in a custom way), use `useSanityPreviewState`:

```ts
const { isPreview, perspective, stegaEnabled } = useSanityPreviewState()
```

| Property | Type | Description |
|---|---|---|
| `isPreview` | `ComputedRef<boolean>` | `true` when Visual Editing / Presentation Tool preview is active |
| `perspective` | `ComputedRef<ClientPerspective>` | Current query perspective (`'published'` or `'previewDrafts'` / `'drafts'`) |
| `stegaEnabled` | `ComputedRef<boolean>` | `true` when stega is both configured and currently active |

All values degrade gracefully to safe defaults (`false` / `'published'`) when visual editing is not configured, so `useSanityPreviewState` is safe to call on any page.

```vue
<template>
  <!-- Only visible to content editors inside Presentation Tool -->
  <div v-if="isPreview" class="preview-banner">
    Previewing {{ perspective }} content
  </div>
</template>

<script setup lang="ts">
const { isPreview, perspective } = useSanityPreviewState()
</script>
```

## Client-Side Navigation Caching with `getCachedData`

By default, every client-side navigation triggers a new Sanity API call. If a user navigates from a product list to a product page, then to another product, and back to the first ‚Äî Sanity is queried every time, even though the data was already fetched.

Nuxt's [`getCachedData`](https://nuxt.com/docs/4.x/api/composables/use-async-data#params) option lets you intercept the fetch and return already-available data instead. If the function returns a value, the network request is skipped entirely.

### Basic usage

```ts
<script setup lang="ts">
const nuxtApp = useNuxtApp()
const route = useRoute()

const { data: product } = await useSanityQuery<Product>(query, params, {
  getCachedData: (key, nuxtApp) =>
    nuxtApp.payload.data[key] ?? nuxtApp.static.data[key],
})
</script>
```

On the first visit to `/products/sneakers`, the SSR payload is hydrated into `nuxtApp.payload.data`. On subsequent client-side navigations back to that URL, `getCachedData` returns the existing payload and the Sanity fetch is skipped.

### Allowing manual refreshes

The `ctx` argument (Nuxt 4+) exposes the `cause` of the data request. Use it to let explicit `refresh()` calls bypass the cache while still serving cached data on navigation:

```ts
getCachedData: (key, nuxtApp, ctx) => {
  if (ctx.cause === 'refresh:manual') return undefined
  return nuxtApp.payload.data[key] ?? nuxtApp.static.data[key]
},
```

### Compatibility with `liveContent`

::warning
`getCachedData` is **not compatible** with `liveContent` unless configured carefully.
::

When `liveContent` is enabled, the module calls `refreshNuxtData(queryKey)` internally when a document changes. This triggers a refetch with `cause: 'refresh:hook'`. If `getCachedData` returns the cached value for this cause, the live update is silently ignored and the page will not reflect the new content.

To support both client-side navigation caching and live content updates, only serve cached data for the initial load and reactive param changes, and let all refresh-type causes pass through:

```ts
getCachedData: (key, nuxtApp, ctx) => {
  // Allow all refreshes (manual, hook-triggered by liveContent, etc.) to re-fetch
  if (ctx.cause !== 'initial' && ctx.cause !== 'watch') return undefined
  return nuxtApp.payload.data[key] ?? nuxtApp.static.data[key]
},
```

| `cause` value | When it fires | With `getCachedData` above |
|---|---|---|
| `'initial'` | First load / SSR hydration | Returns cached SSR payload |
| `'watch'` | Reactive param change (e.g. slug) | Returns cache for that key if available |
| `'refresh:manual'` | Explicit `refresh()` call | Bypasses cache, re-fetches |
| `'refresh:hook'` | `refreshNuxtData()` from `liveContent` | Bypasses cache, re-fetches |

## Stega Encoding Gotchas

When stega is active, Sanity embeds invisible Unicode characters directly into **string values** returned by queries. This is intentional ‚Äî the Visual Editing overlay script reads them to render click-to-edit overlays. However, these characters leak into any context where the raw string is used outside of display:

- **`<title>` and `<meta>` tags** ‚Äî crawlers and social previews see the hidden characters.
- **Sitemap generation** ‚Äî path slugs built from Sanity fields may include them (e.g. with `@nuxtjs/sitemap`).
- **`href` / `src` attributes** ‚Äî URL-encoded invisible characters break links.
- **`JSON.stringify` / external APIs** ‚Äî payloads include the characters and may be rejected.
- **`console.log`** ‚Äî output looks garbled in some terminals.

### The fix: `stegaClean`

`stegaClean` strips all stega characters from a string and returns the clean value. It is part of `@sanity/client`, which is already required when visual editing is configured ‚Äî no extra install needed.

```ts
import { stegaClean } from '@sanity/client/stega'
```

#### SEO meta tags

```ts
<script setup lang="ts">
const { data: product } = await useSanityQuery<Product>(query, { slug: route.params.slug })

// Without stegaClean the invisible characters would appear in <title> and og:* tags
useSeoMeta({
  title: () => stegaClean(product.value?.title),
  ogTitle: () => stegaClean(product.value?.title),
  description: () => stegaClean(product.value?.seo?.description),
})
</script>
```

#### Sitemap with `@nuxtjs/sitemap`

```ts
// server/api/__sitemap__/urls.ts
import { stegaClean } from '@sanity/client/stega'

export default defineSitemapEventHandler(async () => {
  const sanity = useSanity()
  const pages = await sanity.fetch<{ slug: string }[]>(
    `*[_type == "page"]{ "slug": slug.current }`,
  )

  return pages.map(page => ({
    // stegaClean ensures the slug is a plain string with no hidden characters
    loc: `/${stegaClean(page.slug)}`,
  }))
})
```

::tip
`stegaClean` is a no-op when stega is disabled (it returns the input unchanged), so it is safe to call unconditionally in both preview and production contexts.
::

## Summary: When to Use What

| Situation | Recommended composable |
|---|---|
| Public page, no editorial use, maximum CDN performance | `useAsyncData` + `sanity.fetch` |
| Page that may be edited in Presentation Tool | `useSanityQuery` |
| Building custom preview UI / conditional rendering | `useSanityPreviewState` |
| Avoid re-fetching on client-side navigation to already-visited pages | `getCachedData` option |
